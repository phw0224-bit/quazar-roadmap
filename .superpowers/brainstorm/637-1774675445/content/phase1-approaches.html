Phase 1 — 에디터 구현 방식 선택

슬래시 커맨드 + Callout/Toggle 블록을 어떻게 구현할지 결정합니다.

A

@tiptap/suggestion 공식 방식 권장

Tiptap 공식 슬래시 커맨드 패턴. suggestion 확장으로 팝업 위치·키보드 탐색을 처리하고, React 컴포넌트로 메뉴 UI만 담당.

장점

  • 에디터와 완벽 통합 (커서 위치 정확)
  • 키보드(↑↓Enter/Esc) 기본 지원
  • 멘션(@) 추가 시 동일 패턴 재사용
  • 바이브코딩 추가 확장 쉬움

단점

  • 초기 설정 코드 ~80줄
  • 팝업 위치 계산 직접 처리 필요
설치: @tiptap/suggestion, @tiptap/extension-code-block-lowlight,
lowlight, @tiptap/extension-task-list, @tiptap/extension-task-item
B

커스텀 팝업 방식 (floating-ui)

에디터 이벤트를 직접 감지해서 React 포털로 팝업 렌더링. 더 자유로운 UI 커스터마이징 가능.

장점

  • UI 디자인 완전 자유
  • 팝업 위치·애니메이션 세밀 제어
  • Tiptap에 덜 종속

단점

  • 키보드 이벤트 직접 구현
  • 에디터-팝업 동기화 버그 위험
  • 추가 라이브러리 (floating-ui)
설치: @floating-ui/react, + 기타 tiptap 확장들
C

점진적 확장 방식 (슬래시 커맨드 먼저, 블록 나중)

슬래시 커맨드 메뉴 UI/UX를 먼저 완성하고, Callout·Toggle 등 블록은 2단계로 추가.

장점

  • 빠른 첫 결과물 확인
  • 리스크 분산

단점

  • 두 번 작업 필요
  • 구조 변경 가능성
💡 바이브코딩 관점

A 방식이 가장 좋습니다. SlashCommand.js 파일 하나가 "슬래시 커맨드의 모든 로직"을 담고, SlashCommandMenu.jsx가 "팝업 UI만" 담당 → AI가 파일만 보면 바로 이해하고 수정 가능한 구조.
